# 14B Prefill Hybrid MMQ Machine-Search Scope - 2026-07-10

Goal: reproduce the 8B prefill development discipline for Qwen3 14B prefill, while being honest that the 14B fast
path is a different primitive class.

The 8B route reached its useful band by moving from an opaque fast backend toward a machine-owned route envelope:

```text
authority baseline -> extracted specs -> correctness gates -> lifecycle/epoch audit -> bounded timing
-> whole-prefill transfer -> promotion gate
```

For 14B, use the same process, but target a quantized MMQ primitive instead of the 8B fp16 WMMA primitive.

## Classification Target

The intended first useful route is hybrid, not pure generated:

```text
hybrid_machine_search_mmq
  = machine-owned route/spec/search/proof envelope
  + one minimal hand-written MMQ backend atom
```

This mirrors the 8B compromise:

```text
8B hybrid_machine_search
  = machine-owned schedule/spec/search envelope
  + fast backend atom
```

Do not label the 14B path `pure` while it calls a hand-written atom. The route manifest and authority artifacts must
call it hybrid until the backend atom is generated by the compiler.

## Non-Negotiable Boundary

Allowed hand-written surface:

```text
one small parameterized Q4_K/Q8_1 MMQ tile atom
later, if required, one Q6_K/Q8_1 MMQ tile atom
```

Not allowed:

```text
model-name dispatch branches
new 14B-only harness scripts
full-model hard-coded kernels
hidden fallback to direct-packed while claiming MMQ
pure-machine-search claims over hand-written atoms
unmanifested env flags that change promoted behavior
```

The atom must be driven by spec inputs:

```text
role
M/N/K
quant format
tile shape
packed-weight layout
activation Q8_1 layout
output layout
parts/split policy
```

## Why 14B Is Different From 8B

8B prefill searched:

```text
fp16 activations x fp16-resident weights -> fp16 WMMA GEMM
```

14B prefill must search:

```text
fp16 activations -> Q8_1 activation tile
packed Q4_K/Q6_K weights
fused scale/min correction
Q*_K x Q8_1 dot accumulation
output tile ownership
```

The hard part is no longer just LDS/DBUF/waitcnt for fp16 WMMA. It is quantized MMQ lifecycle ownership.

The first atom body has bounded modes:

```text
backend=atom  -> reference-backed executable atom API with lifecycle rows
backend=amd   -> AMD UOp custom-kernel tile atom
backend=amd_warp -> AMD UOp custom-kernel tile atom with one wave reducing 32 Q positions
backend=direct_packed -> bounded comparator using the existing Q4KPrefillRouteSpec direct-packed lowering
backend=amd_warp_batched -> one custom-kernel launch for the whole bounded MxN grid
backend=amd_dot4_batched -> one-launch dot4 inner-loop probe, one output per wave, 8 active lanes
backend=amd_dot4x4_batched -> one-launch dot4 inner-loop probe, four token outputs per wave, 4x8-lane subgroups
```

The AMD path is intentionally narrow: whole-Q4_K-block K tiles only (`k0 % 256 == 0`, `k_groups % 8 == 0`), fp32
output, and no whole-prefill route binding. It records an `amd_uop_hash`; `emitted_binary_hash` is still null because
the harness does not yet capture compiled binary/code-object identity.

Current blocker: representativeness. The one-launch warp atom is now correct and beats the bounded direct-packed
comparator on small synthetic bounded shapes, but this is not whole-prefill authority. The bounded direct-packed
comparator deliberately uses the existing direct-packed math/lowering without forcing the live production `LOCAL`
opts; applying those opts directly in this custom bounded harness rejects with `local is for globals`. Whole-prefill
promotion still requires the canonical 14B smoke/authority path to beat the live direct-packed baseline.

Dot4 probe verdict: correct but not yet better. The first dot4 variants reproduce the key llama inner-loop idea
(`sudot4` over four Q positions, with Q4_K scale/min correction in the dot path), but they do not beat the scalar
batched warp atom in bounded timing. This says the missing llama-level ingredient is not just the dot instruction;
it is tile staging/reuse and output ownership. Keep the dot4 probes as phase evidence, not as a promotion candidate.

## Agent Work Slices

The work is intentionally sliceable:

1. **Llama Route Audit**
   - Read local llama.cpp MMQ source.
   - Produce exact Q4_K/Q8_1 and Q6_K/Q8_1 route anatomy.
   - Output: doc patch or markdown section with source landmarks and constants.

2. **14B Role/Facts Inventory**
   - Use actual GGUF metadata and existing ModelFacts/RoutePlan.
   - Output: JSON/markdown artifact with tensor role, quant, shape, bytes, route-plan decision.

3. **Spec/Reference Gates**
   - Add no-GPU specs/tests for Q4_K x Q8_1 tile math.
   - Output: minimal code/tests in `extra/qk` and `test/unit`.

4. **Harness/Baseline**
   - Run full 14B authority baseline through shared harness.
   - Merge llama comparator or record same-session command.
   - Output: benchmark artifact and short summary.

5. **Atom Boundary Design**
   - Define the minimal hand atom API, lifecycle counters, and route id.
   - Output: design doc patch only unless the spec/reference already exists.

6. **Route Binding Skeleton**
   - After spec exists, add an opt-in route id/policy skeleton that fails loud if the atom is unavailable.
   - Output: tests proving route selection is explicit and no hidden fallback occurs.

## Stop Conditions

Stop and report instead of forcing a patch if:

```text
the route needs a model-name branch
the atom needs full-role hardcoded constants
correctness depends on a giant Tensor intermediate
the harness has to fork into a 14B-only script
the route cannot name whether it is direct-packed, hybrid MMQ, or pure generated
AMD runs become unstable
```

## First Concrete Milestone

The first milestone is not llama parity. It is:

