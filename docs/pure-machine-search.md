# Pure machine search

Roadmap: `docs/pure-machine-search-roadmap.md`.

## The goal

The aim of this fork is "pure machine search": let a search **write** the GPU kernels and **pick** which version to ship, instead of a person hand-writing the fast ones. We do not use tinygrad's built-in autotuner; we built our own search.

We are not there yet. This doc is honest about what is automatic today and what is still hand-written.

## Where we actually are

Two things exist today:

1. **Hand-written hot kernels.** The handful of kernels that matter most for speed are still written by a person. They are written as small code generators (not typed out instruction by instruction), but the design is human.
2. **A machine harness that picks and checks.** Our "search" today takes candidate kernels and settings, runs them through an evaluator, drops the ones that are wrong or already ruled out, and only promotes a candidate if it is correct and measurably faster. It does **not** write new kernels — the loop itself says it "builds no kernels."

So today the machine **chooses and checks**; the person still **writes** the hot kernels. Getting the machine to **write** them is the work that remains.

## The hot kernels: who writes them, who picks them

**Q4_K weight math in decode (matrix × vector).**
- What it is: a hand-written kernel, produced by a small emitter from a hand-designed "lane map" (which GPU lane reads which packed weight). It is not discovered by a search.
- Status: it matches the older hand-tuned kernel within about 0.5% speed, and a selector now picks it automatically for the shapes it supports, with no manual switch. So this is a tidiness win (one clean, automatically-chosen route), not a speed gain over the old kernel.
- We also built a Q6_K version of the same idea and tested it. It was not faster, so it stays off.

**Prefill math (matrix × matrix).**
- What it is: a hand-written assembly kernel. We found the fast settings by measuring options — a software-pipelined version, switched on for the parts that benefit and left off for the one part it slows down — not by an automatic search.
- Status: shipped as the default. It is faster than the previous default at every prompt length we tested, up to 8192 tokens.

**Decode attention.**
- What it is: a hand-written kernel (a split step plus a combine step).
- Status: still the default. We did build a fully generated version on a new low-level backend; it is correct but slower, so it is not shipped. Measurements show attention is now a tiny share of total decode time, so making it faster would barely move the overall number.

## What the model already runs automatically

Most of the model is generated by tinygrad's scheduler with no hand-written kernel: the norms, position encoding, projections, elementwise operations, and the key/value path. Short-context attention also uses a generated version. Only the few hot kernels above are hand-written.

## We tried the automatic path first

We did not hand-write by default. For each hot kernel we first measured the automatic version, built automatic alternatives, and only hand-wrote after those fell short.

- Weight math: the automatic kernel ran at roughly half of the GPU's memory bandwidth because of how it was scheduled (one thread per row, no cross-lane sharing); an automatic int-dot variant was built and lost in the full model before the hand kernel won. [details](archive/qk-mmvq-int-dot-closeout-20260618.md)
- Attention: the reference kernel measured 5–6× faster on its own, and several automatic rewrites were built and rejected before the hand kernel shipped. [details](decode-campaign-final-synthesis-20260623.md)

## Upstream isn't fully automatic either

tinygrad says it generates all kernels with none hand-written. That is true at the whole-kernel level, but not at the instruction level: its renderers hand-code the hottest instructions (the matrix-core / tensor-core op, the workgroup barrier) and splice them into the generated kernels. So the difference is one of degree — upstream hand-codes the hot *instruction*, we hand-code the hot *kernel* — not a clean line. (Citations in the git history.)

## What's left to reach the goal

The remaining work is to make the search actually **write** the kernels, not just pick among hand-written ones:

1. The lane maps, tile shapes, and schedules are still hand-designed. The search would need to generate these itself.
2. Keep the hand-written kernels as the fallback, and only let a generated kernel replace one after it proves it is correct and at least as fast.
3. Spend the effort where it still changes the overall number — the weight math and prefill — not on attention, which is now measured to be low-impact.

## Other hand-written code in the repo

The repo also carries hand-written kernels that the model does **not** run by default: opt-in/research routes (kept off unless a flag is set), measurement tools (used to capture or compare against vendor kernels), and control experiments (used to measure a ceiling, not shipped). None of these are on the default path.
