# Primitive lifecycle search scope - 2026-06-19

Purpose: define the machine-search equivalent for a full performance primitive lifecycle. This is not another kernel
schedule search. It searches over the choices around a primitive: producer placement, data format, consumer family,
routing boundary, quality policy, fallback, and refutation history.

This answers the current question: yes, a lifecycle-level search makes sense, and we have not had it as a formal
system yet. We have been doing it manually through docs and gates. This scope turns it into a table-driven audit layer.

## Why this exists

Kernel search asks:

- which tile, workgroup, unroll, vector width, or schedule wins inside one kernel?

Lifecycle search asks:

- where is the value produced?
- in what format is it stored or forwarded?
- how many consumers reuse it?
- does the producer tax erase the consumer win?
- is the route exact or lossy?
- does the win survive W==D / warm pp / dNLL?
- does the route require an artifact, runtime bridge, graph helper, or compiler feature?
- has an earlier refutation already killed this shape?

That is the right abstraction for the current tinygrad/llama gap. Decode is blocked by a q8/MMVQ activation lifecycle,
not by missing dot4 in isolation. Prefill is blocked by a graph/artifact/codegen lifecycle, not by GEMM math in
isolation.

| layer | current file | job | missing before this scope |

| primitive observability | `extra/qk_primitive_ledger.py`, `bench/qk-primitive-observability/*` | evidence ledger and classifiers | observes outcomes, but does not rank lifecycle candidates |
| scheduler/codegen exhaustion | `bench/amd-schedule-codegen-exhaustion/oracle_matrix.json` | oracle-vs-native capability map | not a candidate generator |

The lifecycle layer consumes the lower layers. It must not replace them.

```json

  "state": "pass_research",
  "role": "Q4_K ffn_gate/up",
  "producer": {
    "placement": "fused RMSNorm/q8 producer",
    "reuse_count": 2
  },
  "format": {
    "activation": "q8_1",
    "weights": "Q4_K",
    "lossy": true
  },
  "consumer": {
    "primitive": "fused gate+up dot4 MMVQ",
    "owner": "external artifact | tinygrad | handwritten ASM"
  },
  "routing": {
    "mode": "research flag | graph node | default | not routed",
    "default_safe": true,
    "artifact_dependency": true,
    "no_hip_runtime_in_process": true
  },
  "quality": {
    "gate": "dNLL <= 0.01",
    "observed_delta": 0.002887
  },
  "score": {
    "speed_gate": "PASS_RESEARCH",
    "ctx_speedup": {"128": 1.063, "512": 1.060},
    "rank": 20
  },
  "evidence": ["docs/...", "bench/..."],
  "blocked_by_refutations": ["..."],
  "next_action": "..."
}
```

Candidate states:

| state | meaning |
|---|---|
| `proposed` | named but not measured |
| `diagnostic` | correctness or partial evidence exists; speed route unresolved |
| `open` | buildable and not yet gated |
| `pass_research` | works behind a research flag but not default-safe as a project policy |
| `pass_strong_policy_gated` | clears speed/quality gates, blocked mainly by artifact/default policy |
| `shipped` | accepted default or supported opt-in |
| `refuted` | measured negative for the named lifecycle |
| `closed` | not worth reopening without new evidence |
| `deferred` | plausible but not buildable until another capability exists |
| `policy_bound` | technically viable but blocked by authority/dependency decision |
| `project_level` | real, but too broad for primitive search |

## Scoring rules

Hard gates first:

1. Correctness or dNLL gate must pass before a candidate can be ranked as viable.
2. A candidate that needs a producer must include producer cost, not just consumer speed.
3. A lifecycle candidate must state reuse count. If reuse is one, activation conversion usually loses.
4. In-model W==D decode or warm pp prefill outranks isolated kernel timing.
5. A default route must have fallback and shape eligibility. Research flags can be looser but must say so.
6. Refutations prune candidates. They are not background context.

Soft ranking:

1. Expected in-model speedup.
2. Amdahl share of the full lifecycle.
3. Quality margin.
4. Dependency/artifact cost.
5. Compiler/runtime surface area.
6. Portability risk.

## What this says about tinygrad's lifecycle story

Decode:

- shipped tinygrad decode is already GPU-bound and not a host-overhead problem;
- q8 artifact lifecycle proves the missing llama-style route is real but small, about `5-6%` in-model;
- native q8 ownership is blocked by producer/codegen/scheduler lifecycle, not by dot4 availability;
- separate-pack, spec-verify, and dot-only rows are pruned.

Prefill:

- pure tinygrad PREFILL_V2 is the clean fallback;
- extracted Tensile proves the mature backend lifecycle can be imported through HCQ without HIP runtime;
- in-model pp512 already clears the strong research gate when all high-share matmul roles are routed;
- the remaining question is policy/hardening, or a project-level native renderer transfer.

## Completion plan

PLS-0: schema and seed rows.

- deliverable: this doc and `extra/qk_lifecycle_search.py`
- gate: JSON artifact validates and ranks existing candidates

PLS-1: refutation memory. **DONE**

- import closed/refuted rows from `qk-machine-search-primitive-rows-20260618.md`
- emit a "do not reopen" report keyed by candidate id
- gate: every live candidate names which refutations it bypasses

Result: `bench/qk-lifecycle-search/refutations.json` has 6 pruning entries and validates that live candidates carry
refutation context.

PLS-2: runner bindings. **DONE**

- bind decode candidates to W==D and dNLL harnesses
- bind prefill candidates to warm pp512/pp1024 and dNLL harnesses
- gate: no candidate can be promoted from `diagnostic` without an in-model runner pointer

Result: `bench/qk-lifecycle-search/runner_bindings.json` binds 6 live/deferred/project rows. These are runner
pointers only; the script does not execute hardware.

PLS-3: policy export. **DONE**

- write accepted research policy records for q8 artifact and Tensile prefill if policy accepts them
- include fallback shape checks and default-off semantics
- gate: no default route changes

Result: `bench/qk-lifecycle-search/policy_exports.json` writes 2 research policy candidates:
`Q8_FFN_HANDWRITTEN=1` and `PREFILL_TENSILE_GEMM=1`. Both remain default-off and decision-required.

PLS-4: generator. **DONE**

- manual/table-driven first: enumerate producer x format x consumer x routing choices only when each axis is legal
- do not use ML or broad BEAM-style search until the table-driven layer proves stale rows are pruned correctly
- gate: generated candidates must cite a speed gate, quality gate, and known refutations before any build

Result: `bench/qk-lifecycle-search/generated_candidates.json` emits generated rows. Legal proposed rows include:

- `decode_q8_sidechannel_native_after_codegen_capability`
- `decode_spec_tcheap_verify_forward` (project-level prerequisite)
- `prefill_tensile_artifact_hardened_shapes`
- `prefill_tensile_native_renderer_transfer`

Rows pruned immediately include:

- `decode_q8_separate_pack_dot4_retry`
- `decode_spec_current_verify_retry`

## Non-goals

- no broad kernel schedule search;
- no new model defaults;
- no bypassing dNLL or exactness gates;
- no treating Tensile extraction as tinygrad ownership;
- no reopening refuted rows with renamed labels;
- no PMU-only verdict without in-model confirmation.

## Next recommendation

Use lifecycle search as the front door for future work:

1. If Claude's LDS/P2 work produces a faster dependency-free prefill candidate, add it as a lifecycle row before
   routing it.
